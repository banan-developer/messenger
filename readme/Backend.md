## Карта бекенда

### Основные маршруты

```text
/login               -> POST: авторизация
/register            -> POST: регистрация
/exit                -> GET: выход из сессии

/api/profile         -> GET own профиль / GET чужой профиль / PUT обновление профиля
/api/profile/avatar  -> POST загрузка аватара

/api/post            -> GET стена своего профиля
                     -> GET стена чужого профиля (query user_id)
                     -> GET данные одного поста для редактирования (query id)
                     -> POST создать пост
                     -> PUT обновить пост (query id)
                     -> DELETE удалить пост (query id)

/api/friend          -> GET список друзей
                     -> GET поиск пользователя (query name)
                     -> GET список друзей другого пользователя (query user_id)
                     -> POST добавить в друзья (query id)

/api/message         -> GET история сообщений с другом (query id, user_id)

/ws                  -> WebSocket соединение для чата
/getUserId           -> GET текущий user_id из сессии
```


---

## Что есть сейчас на бекенде

- session.go — сессии на cookie, `SetUserID`, `GetUserId`, `ClearSessions`
- widdleware.go — middleware `RequireAuth`
- auth.go — регистрация, вход, выход
- api.go — единые точки входа `/api/profile`, `/api/post`, `/api/friend`, `/api/message`
- handlersProfile.go — работа с профилем, стеной, аватаром
- handlersFriend.go — просмотр профиля другого, список друзей другого, поиск
- websocket.go — простой WebSocket чат, хранение коннекций в `map[int]*websocket.Conn`

## Оценка функциональности

### Хорошо
- Базовые CRUD-операции для профиля и стены работают.
- Есть авторизация через сессии.
- Есть базовый друг/чат сценарий: добавление друга, просмотр чужого профиля, 1:1 чат.
- Есть отдельный WebSocket-эндпоинт для мгновенного обмена.

### Плохо / спорно
- main.go регистрирует `/ws` без `RequireAuth` — WebSocket не защищён.
- api.go перегружает один маршрут `/api/post` несколькими разными GET-операциями:
  - получить все посты
  - получить пост для редактирования
  - получить стену другого пользователя
  Это усложняет поддержку и ломает REST-подход.
- `api/friend` тоже смешивает:
  - список друзей
  - поиск
  - список друзей другого пользователя
- `user_id` передаётся из фронта в запросах чат/сообщений, хотя сервер должен брать текущего пользователя из сессии.
- Нет единообразного JSON-ответа/ошибки:
  - иногда метод возвращает просто `http.Error`
  - иногда на успешный PUT нет тела
  - фронт ожидает JSON, но сервер не всегда возвращает
- Нет проверки, что пользователь имеет право редактировать/удалять пост:
  - удаление проверяет `users_id`, но логика не централизована.
- Нет проверки, что можно писать только другу:
  - чат может подхватить любой `friendID`, если URL верный.
- WebSocket-слой:
  - хранит подключения в небезопасной карте без mutex
  - не обрабатывает отключения корректно
  - не поддерживает группы/несколько чатов
- Запросы в базу разбросаны по хендлерам, нет сервисного/репозиторного слоя.
- Нет документации API, нет контрактов, нет тестов.

## Рекомендации: что добавить и улучшить

### Немедленно
- **Защитить `/ws`** через `RequireAuth` и брать `user_id` из сессии.
- **Разделить эндпоинты**:
  - `/api/profile` — `GET /api/profile`, `GET /api/profile/{id}`, `PUT /api/profile`
  - `/api/posts` — `GET /api/posts`, `POST /api/posts`, `GET /api/posts/{id}`, `PUT /api/posts/{id}`, `DELETE /api/posts/{id}`
  - `/api/friends` — `GET /api/friends`, `POST /api/friends/{id}`, `GET /api/friends/search?name=`
  - `/api/chats` — `GET /api/chats`, `GET /api/chats/{id}/messages`
- **Единые JSON-ошибки**:
  - `{ "error": "описание" }`
  - возвращать `415`, `400`, `401`, `403`, `404`, `500` правильно.
- **Не передавать `user_id` в query** для доступа к защищённым действиям.
- **Централизовать доступ к БД**:
  - слой `repository` / `service`
  - меньше SQL в хендлерах
- **Проверять права доступа**:
  - только автор может редактировать пост
  - только друг может писать сообщение
  - профили друзей можно смотреть, остальные — с ограничением
- **Улучшить WebSocket**:
  - hub/manager
  - группы и чат-румы
  - хранить статус онлайна
  - повторные отправки

### Шифрование и безопасность
- На следующем этапе стоит усилить защиту канала связи: HTTPS для всех запросов, безопасные cookie для сессий, защита WebSocket-подключений.
- Для приватных сообщений и чувствительных данных рассматривается переход к шифрованию на уровне передачи и, в перспективе, end-to-end шифрованию.
- Важно также защищать вложения, ограничивать доступ к файлам и хранить пользовательские данные с учётом требований безопасности.

## Итог

### Бек работает, но не стабилен
Сейчас бекенду хватает, чтобы продукт запускался: есть вход, профиль, стена, друзья, чат. Но архитектурно он сделан как прототип:
- смешанные маршруты,
- недостаточная авторизация,
- недостаточная защита WebSocket,
- нет единого API-формата.

### Что важно сделать
1. Разделить маршруты и убрать перегруженные GET.
2. Убирать `user_id` из query, брать его из сессии.
3. Защитить `/ws`.
4. Ввести нормальные JSON-ответы.
5. Построить сервисный слой и убрать SQL из хендлеров.
