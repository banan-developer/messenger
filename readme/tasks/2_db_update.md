### 1. Учебные группы и роли пользователей

Добавляем поля для роли и учебной группы в таблицу пользователей:

```sql
ALTER TABLE `users`
ADD COLUMN `role` VARCHAR(45) NULL,
ADD COLUMN `group_name` VARCHAR(45) NULL;
```

```sql

-- 1. Устанавливаем всем пользователям по умолчанию роль "Студент"
UPDATE `users` 
SET `role` = 'Студент';

-- 2. Присваиваем номер группы 1442 известным участникам проекта (Ильназу и Эле)
UPDATE `users` 
SET `group_name` = '1442' 
WHERE `id` IN (21, 34);

-- 3. (Пример) Если вам нужно назначить кого-то из тестовых пользователей преподавателем:
-- Например, сделаем "Тестового пользователя" (ID 32) преподавателем и уберем ему группу
UPDATE `users` 
SET `role` = 'Преподаватель', 
    `group_name` = NULL 
WHERE `id` = 32;
```

### 4.2 Групповые чаты. Код для взаимодействия

Теперь рассмотрим основные SQL-запросы (с примерами для ваших тестовых пользователей: Ильназ `id=21`, Эля `id=34`, Муртуз `id=25`).

#### 1. Проверка: есть ли уже личная переписка между пользователями?

Прежде чем создать личный чат между Ильназом (21) и Элей (34), бекенд должен проверить, не существует ли он уже.

```sql
SELECT c.id
FROM `chats` c
JOIN `users_has_chats` uhc1 ON c.id = uhc1.chats_id
JOIN `users_has_chats` uhc2 ON c.id = uhc2.chats_id
WHERE c.is_group = 0 
  AND uhc1.users_id = 21 
  AND uhc2.users_id = 34;

```

Если запрос возвращает ID чата (например, 13), мы просто открываем этот чат. Если возвращает пустоту — создаем новый.

#### 2. Создание новой личной переписки (тет-а-тет)

Если чат не найден, выполняем транзакцию из двух шагов:

```sql
-- Шаг 1: Создаем сам чат (title пустой, is_group = 0)
INSERT INTO `chats` (`is_group`, `title`, `avatar_url`) 
VALUES (0, NULL, NULL);

-- В коде Golang вы получаете ID созданного чата (например, это ID 50).
-- Шаг 2: Добавляем обоих пользователей в этот чат
INSERT INTO `users_has_chats` (`users_id`, `chats_id`) 
VALUES (21, 50), (34, 50);

```

#### 3. Создание групповой беседы

Здесь логика похожая, но мы указываем название группы, и участников может быть больше двух.

```sql
-- Шаг 1: Создаем группу
INSERT INTO `chats` (`is_group`, `title`, `avatar_url`) 
VALUES (1, 'Проект по ООП', '/static/uploads/groups/oop.png');

-- Предположим, новый чат получил ID = 51.
-- Шаг 2: Добавляем Ильназа, Элю и Муртуза в группу
INSERT INTO `users_has_chats` (`users_id`, `chats_id`) 
VALUES (21, 51), (34, 51), (25, 51);

```

#### 4. Получение списка участников чата (для группы)

Чтобы вывести на экран информацию о группе и список кто в ней состоит (чат 51):

```sql
SELECT u.id, u.name, u.avatar_url, u.role
FROM `users` u
JOIN `users_has_chats` uhc ON u.id = uhc.users_id

```

#### 6. Получение списка ВСЕХ чатов пользователя (для бокового меню)

Это самый сложный и важный запрос для фронтенда. Когда Ильназ (21) открывает раздел "Сообщения", ему нужно показать и группы, и личные диалоги. При этом для групп названием будет `title`, а для личных чатов — **имя собеседника**.

```sql
SELECT 
    c.id AS chat_id,
    c.is_group,
    -- Если это группа, берем её название. Если личный чат — имя собеседника.
    IF(c.is_group = 1, c.title, u_partner.name) AS chat_name,
    -- То же самое для аватарки
    IF(c.is_group = 1, c.avatar_url, u_partner.avatar_url) AS chat_avatar,
    -- Получаем текст последнего сообщения (опционально, для превью)
    (SELECT text FROM messeges m WHERE m.chats_id = c.id ORDER BY m.created_at DESC LIMIT 1) AS last_message
FROM `chats` c
JOIN `users_has_chats` uhc_me ON c.id = uhc_me.chats_id
-- Левое соединение (LEFT JOIN), чтобы найти собеседника только для личных чатов
LEFT JOIN `users_has_chats` uhc_partner ON c.id = uhc_partner.chats_id AND uhc_partner.users_id != 21
LEFT JOIN `users` u_partner ON uhc_partner.users_id = u_partner.id
WHERE uhc_me.users_id = 21
GROUP BY c.id;

```

Этот запрос магическим образом отформатирует список так, как показано на вашем макете экрана "Сообщения" (где вперемешку идут и "Группа 1442", и личный чат "Алексей Петров").

Когда пользователь открывает конкретный диалог (например, нажимая на карточку чата из списка), вашему бекенду необходимо выполнить два действия: получить информацию для «шапки» чата (имя собеседника/название группы и аватар) и загрузить саму историю сообщений.

Вот SQL-запросы для реализации макета экрана диалога (Рисунок 3). Для примера предположим, что Ильназ (`id = 21`) открывает чат с `chats_id = 13`.

#### 7. Получение данных для "шапки" текущего чата

Этот запрос определяет, как будет называться чат в заголовке и какая картинка будет отображаться (зависит от того, группа это или личный диалог).

```sql
SELECT 
    c.id AS chat_id,
    c.is_group,
    -- Если группа - берём её название, если личный диалог - имя собеседника
    IF(c.is_group = 1, c.title, u_partner.name) AS chat_name,
    -- Аналогично для аватарки
    IF(c.is_group = 1, c.avatar_url, u_partner.avatar_url) AS chat_avatar,
    -- Для групп можно сразу посчитать количество участников
    (SELECT COUNT(*) FROM `users_has_chats` WHERE chats_id = 13) AS members_count
FROM `chats` c
-- Ищем собеседника (если это личный чат), исключая самого себя (21)
LEFT JOIN `users_has_chats` uhc_partner ON c.id = uhc_partner.chats_id AND uhc_partner.users_id != 21
LEFT JOIN `users` u_partner ON uhc_partner.users_id = u_partner.id
WHERE c.id = 13
LIMIT 1;

```

#### 8. Получение истории сообщений

Этот запрос вытягивает сами сообщения из таблицы `messeges`, присоединяя данные отправителя из таблицы `users`, чтобы фронтенд мог отобразить аватарку и имя возле каждой реплики.

Так как в дампе базы данных (в переписке Ильназа и Эли, сообщения 262–272) уже обсуждалась проблема того, что «нет смысла все сообщения сразу подгружать» и «нужно пагинацию... сделать», в этот запрос сразу заложена основа для пагинации (`LIMIT` и `OFFSET`).

```sql
SELECT 
    m.id AS message_id,
    m.text,
    m.created_at,
    m.attachment_url,
    m.is_read,
    m.from_id,
    u.name AS sender_name,
    u.avatar_url AS sender_avatar
FROM `messeges` m
JOIN `users` u ON m.from_id = u.id
WHERE m.chats_id = 13
ORDER BY m.created_at DESC
LIMIT 50 OFFSET 0; 

```

**Как с этим работать на фронтенде/бекенде:**

1. **Сортировка `ORDER BY m.created_at DESC**`: Мы сортируем от новых к старым. Так проще реализовывать подгрузку сообщений при скролле вверх (бесконечный скролл/пагинация). На самом клиенте массив перед отрисовкой обычно переворачивают (reverse), чтобы новые сообщения оказались внизу.
2. **Пагинация**: `LIMIT 50 OFFSET 0` загрузит последние 50 сообщений. Когда пользователь прокрутит чат наверх, бекенд должен сделать запрос `LIMIT 50 OFFSET 50` для получения следующей порции.
3. **Отрисовка своих/чужих**: На фронтенде вы просто сравниваете свой собственный ID (например, 21) с полем `from_id` каждого сообщения. Если они совпадают — рендерите сообщение справа («Исходящее сообщение (Вы)»). Если не совпадают — слева («Входящее сообщение (Собеседник)»).
